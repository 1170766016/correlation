# 工业级跨代理 HTTP 流量追踪与分析系统设计文档

## 一、 业务背景与核心目标

### 1.1 业务痛点
在工厂产线网络环境中，约有 15000 个客户端与后端的 MES 系统进行高并发交互，中间经过负载均衡/代理服务器（SNAT/DNAT 转换）。传统基于 TCP 四元组的网络排查方式失效，无法串联“客户端 $\to$ 代理”和“代理 $\to$ 服务器”的完整数据流。

### 1.2 系统演进目标（基于 `detech.py` 的重构）
根据最新指示，系统操作流将发生根本性改变：
- **废弃“先过滤后入库”**：不再通过界面输入 BPF 规则把数据载入内存（这会导致超大文件 OOM 和数据截断）。
- **全量无状态解析入库**：对 200GB+ 的 PCAP 文件进行**流式读取**，提取所有报文的关键元数据（丢弃 200KB 原始载荷以省空间），直接批量灌入 SQL Server。
- **构建完整报文回路**：在数据库中，通过多维度的特征（哈希、四元组）进行事后筛选与关联，真正构建出**“前端请求 $\to$ 后端请求 $\to$ 后端响应 $\to$ 前端响应”**的完整闭环。
- **智能化故障诊断**：依靠全量元数据，在数据库层面通过 SQL 查询直接诊断重复、丢包、高延迟等问题。

## 二、 核心架构设计：构建完整报文回路

要想构建完整的“四段式”报文回路，需要结合**应用层指纹**和**传输层状态**：

### 2.1 步骤一：连接前端与后端请求 (Request 跨段穿透)
**挑战**：IP 和端口被代理改变。
**方案**：**基于载荷哈希指纹 (`payload_hash`)**
1. 提取 HTTP POST 的 Body（200KB JSON）计算 MD5。
2. 数据库自关联：寻找 `payload_hash` 相同且时间相近的两条记录。
3. **结果**：时间早的为“前端请求”，时间晚的为“后端请求”。此时，我们成功获取了**前端 TCP 连接的四元组**和**后端 TCP 连接的四元组**。

### 2.2 步骤二：连接请求与响应 (Request $\to$ Response)
**挑战**：HTTP 返回包的 Body 是响应结果，MD5 变了，无法用 `payload_hash` 关联。
**方案**：**基于 TCP 四元组与时序追踪**
1. 针对“前端请求”，利用其所在的**前端 TCP 四元组**，在数据库中寻找该连接上、时间紧随其后的 HTTP 返回包，即为“前端响应”。
2. 针对“后端请求”，利用其所在的**后端 TCP 四元组**，在数据库中寻找紧随其后的 HTTP 返回包，即为“后端响应”。

通过这两步，一条请求跨越 NAT 转换的完整生命周期（请求进 $\to$ 请求出 $\to$ 响应进 $\to$ 响应出）被完全闭环。

## 三、 数据库设计规范

为了支撑全量入库和事后的极速分析，`dbo.packets` 字段必须精简且具备极高的索引效率。

| 字段名称 | 数据类型 | 说明 | 作用 |
| :--- | :--- | :--- | :--- |
| `id` | `BIGINT IDENTITY` | 自增主键 | 支撑亿级数据，代替 INT |
| `pcap_file` | `VARCHAR(255)` | 抓包文件名 | 文件级查重 |
| `timestamp` | `DATETIME2(6)` | 抓包时间戳 | 报文绝对时序 |
| `mac_src`/`dst`| `VARCHAR(20)` | MAC 地址 | 二层排查（按需留存）|
| `ip_src`/`dst` | `VARCHAR(45)` | 源/目的 IP | TCP 四元组要素 |
| `sport`/`dport`| `INT` | 源/目的 端口 | TCP 四元组要素 |
| `flag` | `VARCHAR(30)` | TCP 标志位 | 用于识别 TCP 重传、RST 断连等异常 |
| `seq`/`ack` | `BIGINT` | TCP 序列号 | 用于深度 TCP 丢包和重传判定 |
| `len` | `INT` | 载荷长度 | |
| `post_url` | `NVARCHAR(2000)`| HTTP URL | 事件类型 |
| `client_time` | `DATETIME2(3)` | 客户端时间 | 从 JSON 提取的 `sendTime`，业务对齐关键 |
| `payload_hash`| `VARCHAR(32)` | 请求 Body MD5 | **构建报文回路的破局点** |
| `frame_number`| `BIGINT` | PCAP 帧序号 | 定位原始报文以提取 200KB 详情 |

**核心索引**：
1. `IX_packets_payload_hash (payload_hash, timestamp)`：极速配对前后端请求。
2. `IX_packets_tcp_session (ip_src, ip_dst, sport, dport, timestamp)`：极速配对同一 TCP 连接内的请求与响应。

## 四、 问题分析与定界模型（基于数据库 SQL 的筛选体系）

入库完成后，我们可以通过 SQL 筛选快速定位以下问题：

### 4.1 代理服务器丢包 (Proxy Drop)
**特征**：有前端请求，但查不到同一 `payload_hash` 的后端请求。
**SQL 逻辑**：查询包含 `payload_hash` 的所有报文，若 `COUNT(*) = 1` 且目标 IP 属于代理 VIP，说明报文进了代理但没有发往后端服务器。

### 4.2 Web 服务器处理丢包或超时 (Server Timeout)
**特征**：前端请求、后端请求均存在，但后端 TCP 连接上迟迟没有对应的 HTTP 响应包，最后可能伴随 TCP RST 或 FIN。
**SQL 逻辑**：定位到后端请求后，按其 TCP 四元组向后查，发现直到连接关闭（`flag` 含 RST/FIN）也没有大的返回包。

### 4.3 代理转发延迟过高 (Proxy Latency)
**特征**：前后端请求都存在，但时间差极大。
**SQL 逻辑**：`DATEDIFF(ms, 前端请求.timestamp, 后端请求.timestamp) > 500ms`。

### 4.4 客户端重复发包 / 重传 (Client Duplication)
**特征**：同一个 `payload_hash` 出现了 3 次及以上。如果是应用层重复发包，对应的 IP/端口可能不同；如果是 TCP 层重传，则 `seq` 会相同。
**SQL 逻辑**：`GROUP BY payload_hash HAVING COUNT(*) >= 3`。

## 五、 方案边界、致命风险与改造细节

基于 `detech.py` 现状，必须彻底改造以下环节以防方案破产：

### 🔴 风险 1：全量入库引发 OOM（必须改写主流程）
**当前代码**：一次性读取文件 $\to$ 全量加载到内存做 UI 呈现和过滤 $\to$ 再转 List 入库。对 20GB+ 文件必定崩溃。
**改造细节**：废弃 `do_filter` 中的 `filtered_pkts` 内存缓存。将解析与入库合并为一条**流式处理管道 (Streaming Pipeline)**：使用 `PcapReader` 逐包读取，单包解析生成 Tuple 后立即放入队列，攒够 5000 条直接触发 `fast_executemany` 写入数据库，释放内存。内存占用将永远保持在极低水平（几十 MB）。

### 🔴 风险 2：TCP 分段导致 HTTP 提取失败
**细节**：200KB 的业务 JSON 会被切成 140 多个 TCP 包。直接读第一个包算 MD5 会导致哈希错乱，整个回路无法闭环。
**改造细节**：`detech.py` 在流式解析时，必须引入 TCP 流重组（如使用 Scapy 的 `TCPSession` 追踪流，或者底层换用 `pyshark`/`tshark -T json`），确保拼接出完整的 HTTP 报文后再计算 MD5。

### 🟡 风险 3：UI 交互重构
**细节**：既然不再是在界面上“先过滤再入库”，UI 的逻辑应当转变为：
1. **任务面板**：选择大 PCAP 文件 $\to$ 点击“解析并全量入库” $\to$ 显示进度条。
2. **分析面板**：提供条件输入框（如输入 `client_time` 或 URL） $\to$ 点击“查询” $\to$ 程序去 SQL Server 检索，并将拼装好的“前端请求、后端请求、响应时间”等回路信息展示在表格中。
3. **详情面板**：双击一条回路记录 $\to$ 利用 `frame_number` 从本地 PCAP 精准切片，读取那 200KB JSON 供人工查阅。

### 🟢 风险 4：HTTPS 加密 / 压缩
**细节**：本回路追踪机制高度依赖明文 HTTP Body。若启用了 HTTPS 或 GZIP，必须在抓包环境（解密后镜像）或解析器层面预先还原明文，否则 `payload_hash` 无法计算。

## 六、 总结落地步骤

1. **改造 `detech.py` 入库流**：移除内存列表，改为流式读取 + TCP 重组 + 批量数据库 Insert 管道。
2. **完善 `db_manager.py` 分析逻辑**：写入 4 个核心排查 SQL（寻丢包、寻高延迟、寻回路）。
3. **改造 Tkinter UI**：重塑为“大数据后台处理”+“数据库级前端探查”的现代排查工具界面。
