# 数据库存储过程部署与使用指南

## 架构变更说明

本次重构将原本散落在 `detech.py` 的数据库操作拆分为三层：

```
detech.py (UI 层)
    ↓ 调用
db_manager.py (Python 数据访问层 - PcapDbManager)
    ↓ 调用
SQL Server 存储过程 (业务逻辑层 - db_schema.sql)
    ↓ 操作
packets 表 (表结构定义 - db_init.sql)
```

## 文件清单

| 文件 | 作用 | 执行顺序 |
|---|---|---|
| `db_init.sql` | **纯 DDL**：建表 + 索引（表结构权威文档） | 1. 全新部署时先执行 |
| `db_schema.sql` | 4 个存储过程（业务逻辑） | 2. 必须执行 |
| `db_manager.py` | Python 数据访问层 | 不执行，被 detech.py 导入 |
| `detech.py` | UI 层 | 程序入口 |

## 部署步骤

### 1. 部署表结构（在服务器上执行一次）

```cmd
sqlcmd -S localhost -E -d pcap_db -i db_init.sql
```

执行后会打印建表成功 + 索引创建 + 表结构验证结果。

### 2. 部署存储过程（在服务器上执行一次）

```cmd
sqlcmd -S localhost -E -d pcap_db -i db_schema.sql
```

## 为什么批量插入不走存储过程

pyodbc 的 `fast_executemany=True` 对 `INSERT ... VALUES(?,?,...)` 是**一次性 RPC 发送整批参数**，N 行只需 1 次网络往返。

如果改成调用存储过程 + `executemany`，pyodbc 会变成**每行 1 次 RPC 调用**，N 次网络往返。对 18 万行数据，性能下降 **5-10 倍**。

因此：
- **业务逻辑**（建表/ALTER/索引/查重/删除）→ 存储过程（维护跟踪重点）
- **批量插入** → 保留 `executemany + INSERT VALUES`（性能关键路径）

## 部署步骤

### 1. 部署存储过程（在服务器上执行一次）

**方式一：SSMS 图形界面**

1. 用 SSMS 连接到 `localhost`
2. 选择 `pcap_db` 数据库
3. 文件 → 打开 → `db_schema.sql`
4. 执行（F5）

**方式二：sqlcmd 命令行**

```cmd
sqlcmd -S localhost -E -d pcap_db -i db_schema.sql
```

执行成功会看到 4 条记录：
```
sp_check_pcap_duplicate   SQL_STORED_PROCEDURE
sp_delete_pcap_by_file    SQL_STORED_PROCEDURE
sp_ensure_pcap_table      SQL_STORED_PROCEDURE
sp_get_pcap_stats         SQL_STORED_PROCEDURE
```

### 2. 验证部署

```sql
EXEC sp_ensure_pcap_table 'packets';
EXEC sp_get_pcap_stats 'packets';
```

第一条确保表结构；第二条返回总行数、文件数、最近入库时间。

### 3. 运行程序

正常启动 `detech.py` 即可。程序会自动：
- 检测存储过程是否已部署（未部署会提示但不会崩溃，走兜底内联 SQL）
- 调用 `sp_ensure_pcap_table` 维护表结构
- 调用 `sp_check_pcap_duplicate` 查重
- 调用 `sp_delete_pcap_by_file` 覆盖旧数据
- 用 `executemany + INSERT VALUES` 批量插入

## 存储过程清单

### 1. sp_ensure_pcap_table

确保表结构存在并维护索引。幂等，可重复调用。

```sql
EXEC sp_ensure_pcap_table 'packets';
```

功能：
- 表不存在则创建（含 16 列）
- 自动补齐缺失列（兼容旧版表）
- 维护非聚集索引 `IX_pcap_file`、`IX_pcap_hash`
- 移除旧版唯一索引 `UQ_packet_hash`

### 2. sp_check_pcap_duplicate

检查抓包文件是否已入库。返回状态码：

| status | 含义 | 调用方动作 |
|---|---|---|
| 0 | 未重复 | 继续入库 |
| 1 | 内容指纹相同 | 跳过导入 |
| 2 | 同名不同内容 | 先删除再入库 |

```sql
DECLARE @st INT, @ef VARCHAR(255);
EXEC sp_check_pcap_duplicate 'packets', 'abc123hash', '525.pcap', @st OUTPUT, @ef OUTPUT;
SELECT @st AS status, @ef AS existing_file;
```

### 3. sp_delete_pcap_by_file

按文件名删除旧数据（覆盖导入场景）。

```sql
EXEC sp_delete_pcap_by_file 'packets', '525.pcap';
-- 返回删除行数
```

### 4. sp_get_pcap_stats

查询表统计信息（运维监控用）。

```sql
EXEC sp_get_pcap_stats 'packets';
-- 返回 total_rows, file_count, latest_insert_time
```

## 维护与跟踪

### 修改表结构

只需修改 `sp_ensure_pcap_table` 中的 DDL，重新部署 `db_schema.sql` 即可。下次程序运行时自动应用。

### 查看入库历史

```sql
-- 查所有入库文件
SELECT DISTINCT pcap_file, pcap_hash, MIN(insert_time) AS first_import, MAX(insert_time) AS last_import
FROM packets
GROUP BY pcap_file, pcap_hash
ORDER BY last_import DESC;

-- 查某文件报文统计
SELECT pcap_file, COUNT(*) AS rows, MIN(timestamp) AS start_time, MAX(timestamp) AS end_time
FROM packets
WHERE pcap_file = '525.pcap'
GROUP BY pcap_file;
```

### 性能监控

```sql
-- 查看存储过程执行统计
SELECT
    OBJECT_NAME(object_id) AS proc_name,
    execution_count,
    total_elapsed_time / 1000.0 AS total_sec,
    avg_elapsed_time / 1000.0 AS avg_sec
FROM sys.dm_exec_procedure_stats
WHERE database_id = DB_ID('pcap_db')
ORDER BY total_elapsed_time DESC;
```

## 故障排查

### 程序提示"未检测到存储过程"

执行 `db_schema.sql` 部署即可。程序会走兜底内联 SQL 模式，功能正常但失去集中维护优势。

### 调用存储过程报错

检查：
1. 当前 Windows 账号对 `pcap_db` 有 `db_owner` 或 `db_ddladmin` 角色
2. 存储过程存在：`SELECT name FROM sys.objects WHERE name LIKE 'sp_%' AND type = 'P';`
3. 表存在：`SELECT OBJECT_ID('dbo.packets', 'U');`

### 批量插入失败

存储过程不影响批量插入路径。如果报 2100 参数超限，检查 `pyodbc` 版本：

```cmd
pip show pyodbc
```

要求 ≥ 4.0.19 才支持 `fast_executemany`。`db_manager.py` 已内置兜底：参数超限时自动切到 131 行/批。
